# E2EE — Ende-zu-Ende-Verschlüsselung

> Stand: 24.03.2026 — Phase 2/3 implementiert (Cross-Signing aktiv)

---

## Architektur-Entscheidung: Option C

```
Matrix-Homeserver (tuwunel)
        ↓  encrypted events (m.room.encrypted)
Go Appservice (goolm — Pure-Go Olm/Megolm)
        ↓  Klartext-Nachrichten via NATS
Python Agent Bridge  ←→  Next.js Browser (initRustCrypto aktiv)
```

**Go Appservice** übernimmt alle Crypto-Operationen für den Bot:
- Empfangen: `m.room.encrypted` → OlmMachine.Decrypt() → Klartext
- Senden: Klartext → Machine.Encrypt() → `m.room.encrypted`
- Key Sharing: `m.key.verification.*`, To-Device Events
- Cross-Signing Bootstrap: `ensureCrossSigning()` bei Start

**Next.js Browser** hat eigene E2EE-Schicht (Phase 2):
- `initRustCrypto()` aktiv — vodozemac (WASM) für User-Nachrichten
- Cross-Signing via `useCrossSigning` Hook + `CrossSigningSetup`-Komponente

**Python Agent Bridge** empfängt nur Klartext. Kein libolm/vodozemac nötig.

## Warum Option C (nicht A oder B)

| Option | Beschreibung | Problem |
|---|---|---|
| A — Python E2EE | matrix-nio[e2e] | libolm.dll auf Windows nicht verfügbar |
| B — Browser E2EE | initRustCrypto() im Browser | Bot-Antworten vom Server-Prozess kommen nicht durch |
| **C — Go E2EE** | mautrix-go + goolm | **Kein CGO, Pure-Go, Windows-kompatibel** ✅ |

## Kryptografie-Schicht (2026)

```
Olm     → 1:1 Schlüsselaustausch (X3DH)
Megolm  → Gruppen-Verschlüsselung (AES-256-GCM + Double Ratchet)
goolm   → Pure-Go Reimplementierung beider Algorithmen (Build-Tag: -tags goolm)
```

### goolm vs vodozemac

| | vodozemac (Rust) | goolm (Go) |
|---|---|---|
| Olm/Megolm | ✅ | ✅ identisch |
| PQXDH (Post-Quantum ML-KEM-768) | ✅ | ❌ noch nicht |
| CGO nötig | ja (Rust→C Bridge) | nein — Pure-Go |
| Windows Dev | MinGW/GCC nötig | funktioniert direkt |
| Production Linux | bevorzugt | ausreichend |

**Warum goolm und nicht vodozemac:**
Auf Windows braucht vodozemac CGO (MinGW/GCC Toolchain). Im Dev-Stack (devstack.ps1)
nicht praktikabel. goolm ist funktional identisch für Olm+Megolm.
Für Production auf Linux: Portierungsplan sieht vodozemac via CGO vor (siehe `10-portierung.md`).

### Crypto-Store (SQLite)

```go
// internal/crypto/machine.go
import (
  "database/sql"
  _ "modernc.org/sqlite"         // Pure-Go SQLite
  "go.mau.fi/util/dbutil"
  "maunium.net/go/mautrix/crypto"
)

rawDB, _ := sql.Open("sqlite", dbPath+"?_foreign_keys=on&_txlock=immediate")
rawDB.SetMaxOpenConns(1)         // SQLite: single writer
db, _ := dbutil.NewWithDB(rawDB, "sqlite3")

cryptoStore := crypto.NewSQLCryptoStore(
  db,
  dbutil.NoopLogger,
  client.UserID.String(),
  "APPSERVICE",
  pickleKey,
)
// Kein separates Upgrade() — intern in NewSQLCryptoStore
```

### OlmMachine Setup

```go
// goolm Build-Tag aktiviert Pure-Go Crypto
olmMachine := crypto.NewOlmMachine(client, &zlog, cryptoStore, stateStore)
olmMachine.SendKeysMinTrust = id.TrustStateUnset  // sendet Keys an alle Geräte (Dev+Cross-Signing)
olmMachine.Load(ctx)
olmMachine.ShareKeys(ctx, -1)
```

**Hinweis zu `SendKeysMinTrust = TrustStateUnset`:**
Standardmäßig würde OlmMachine Megolm-Keys nur an verifizierte Geräte senden.
`TrustStateUnset` erlaubt das Senden an alle Geräte — nötig, da der Go Appservice
selbst Cross-Signing-verifiziert ist, aber nicht alle Peers aktiv verifiziert.

## Go Appservice — Cross-Signing Bootstrap (C-3)

### ensureCrossSigning()

Implementiert in `internal/crypto/machine.go`. Wird nach `olmMachine.Load()` aufgerufen.

**Erster Start (keine Seeds-Datei vorhanden):**
```go
// 1. Cross-Signing-Keys generieren und hochladen
err := olmMachine.GenerateAndUploadCrossSigningKeys(ctx, nil, "")
// 2. Eigenes Gerät signieren
err = olmMachine.SignOwnDevice(ctx)
// 3. Eigenen Master Key signieren
err = olmMachine.SignOwnMasterKey(ctx)
// 4. Seeds persistieren
seeds := CrossSigningSeeds{
  MasterKey:      olmMachine.GetCrossSigningKeys().MasterKey.Seed,
  SelfSigningKey: olmMachine.GetCrossSigningKeys().SelfSigningKey.Seed,
  UserSigningKey: olmMachine.GetCrossSigningKeys().UserSigningKey.Seed,
}
writeJSON("<dbDir>/cross_signing_seeds.json", seeds, 0o600)
```

**Neustart (Seeds-Datei existiert):**
```go
// 1. Seeds laden und importieren
seeds := readJSON("<dbDir>/cross_signing_seeds.json")
err := olmMachine.ImportCrossSigningKeys(seeds)
// 2. Signaturen erneuern
err = olmMachine.SignOwnDevice(ctx)
err = olmMachine.SignOwnMasterKey(ctx)
```

### Seeds-Datei

- **Pfad:** `<dbDir>/cross_signing_seeds.json` (dbDir = MATRIX_CRYPTO_DB_PATH-Verzeichnis)
- **Berechtigungen:** 0o600 (nur Owner lesbar)
- **Inhalt:** Master Key Seed, Self-Signing Key Seed, User-Signing Key Seed (Base64)
- Die Seeds sind dauerhaft — Cross-Signing-Identität bleibt über Neustarts erhalten

## Browser (Next.js) — E2EE Status

`initRustCrypto()` ist seit Phase 2 **aktiv**. Der Browser nutzt vodozemac (WASM) für User-Nachrichten.

```typescript
// src/lib/matrix/client.ts
await client.initRustCrypto();
client.getCrypto()!.globalBlacklistUnverifiedDevices = false;
```

- `globalBlacklistUnverifiedDevices = false`: Sendet auch an unverifizierte Geräte (Dev-Modus, D-1 offen)
- Key Store: IndexedDB (pro User, pro Browser) — **aktiv**
- Cross-Signing-Verifikation via `CrossSigningSetup`-Komponente (QR-Code + SAS-Emoji-Fallback)
- `useCrossSigning` Hook: überwacht `isCrossSigningReady()`, behandelt `CryptoEvent.VerificationRequestReceived`

## E2EE Rollout Strategie

### Phase 1 — Dev/Testing (abgeschlossen)
- Go Appservice mit `MATRIX_E2EE_ENABLED=false` → unverschlüsselte Räume
- Kein E2EE-Setup nötig
- Fokus auf Funktionalität

### Phase 2 — Interne Tests (abgeschlossen)
- `MATRIX_E2EE_ENABLED=true` im devstack
- Go Appservice: goolm + SQLite Crypto Store
- Python: Klartext via NATS (keine Änderung)
- Browser: Sendet unverschlüsselt (Klartext vom User) — noch ohne initRustCrypto

### Phase 3 — Cross-Signing aktiv (aktuell)
- Browser: `initRustCrypto()` aktiv (WASM vodozemac) ✅
- Go Appservice: `ensureCrossSigning()` Bootstrap ✅ — Seeds persistiert in cross_signing_seeds.json
- Browser: Cross-Signing via `CrossSigningSetup`-Komponente (QR + SAS) ✅
- MSC4153: Go Appservice ist cross-signing-verifiziert ✅
- **Offen (D-1):** `globalBlacklistUnverifiedDevices` noch auf `false` — für Production auf `true` setzen

### Phase 4 — Production
- `globalBlacklistUnverifiedDevices = true` im Browser (D-1)
- Key Backup implementieren (C-8) — Megolm Key Backup für neue Geräte
- PQXDH via vodozemac im Go Appservice (Portierung auf Production Linux)
- Authenticated Media (MSC3916) — `allow_legacy_media` Workaround entfernen

## MSC4153 — Implementiert

Ab April 2026 senden major Clients (Element Web, Element X) E2EE To-Device Nachrichten
nur noch an **kryptografisch verifizierte Geräte** (Cross-Signing).

**Status:** Implementiert (C-3)
- Go Appservice: `ensureCrossSigning()` sorgt für automatischen Bootstrap bei erstem Start
- Seeds werden persistiert → Cross-Signing-Identität überlebt Neustarts
- `olmMachine.SendKeysMinTrust = id.TrustStateUnset` → Keys werden an alle Geräte gesendet
- Browser: `CrossSigningSetup`-Komponente für QR-Verifikation zwischen Web-Gerät und Go Appservice

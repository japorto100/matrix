## Architektur-Entscheidung: Option C — Go als E2EE-Endpunkt

```
Matrix Homeserver (Tuwunel)
        ↓ encrypted events (m.room.encrypted)
Go Appservice (goolm — Pure-Go Olm/Megolm)
        ↓ Klartext-Nachrichten via NATS
Python Backend (bridge → agent)        ←→  Next.js Browser (initRustCrypto aktiv)
```

**Go Appservice** uebernimmt alle Crypto-Operationen fuer die Bot-Identitaet:
- Empfangen: `m.room.encrypted` → `OlmMachine.Decrypt()` → Klartext
- Senden: Klartext → `Machine.Encrypt()` → `m.room.encrypted`
- Key Sharing: `m.key.verification.*`, To-Device Events
- Cross-Signing Bootstrap: `ensureCrossSigning()` bei Start

**Next.js Browser** hat eigene E2EE-Schicht (Phase 2/3):
- `initRustCrypto()` aktiv — vodozemac (WASM) fuer User-Nachrichten
- Cross-Signing via `useCrossSigning` Hook + `CrossSigningSetup`-Komponente

**Python Backend** empfaengt nur Klartext via NATS. Kein libolm/vodozemac noetig.
matrix-nio wurde in exec-05 komplett entfernt — Python kennt kein Matrix-Protokoll mehr.

| Option | Beschreibung | Problem |
|---|---|---|
| A — Python E2EE | matrix-nio[e2e] | libolm-Build auf Windows broken; doppelte Cross-Signing-Verwaltung |

## Kryptografie-Schicht (2026)

```
Olm     → 1:1 Schluesselaustausch (X3DH)
Megolm  → Gruppen-Verschluesselung (AES-256-GCM + Double Ratchet)
goolm   → Pure-Go Reimplementierung beider Algorithmen (Build-Tag: -tags goolm)
```

### goolm vs vodozemac

| | vodozemac (Rust) | goolm (Go) |
|---|---|---|
| Olm/Megolm | ✅ | ✅ identisch |
| PQXDH (Post-Quantum ML-KEM-768) | ✅ | ❌ noch nicht |
| CGO noetig | ja (Rust→C Bridge) | nein — Pure-Go |
| Windows Dev | MinGW/GCC noetig | funktioniert direkt |
| Production Linux | bevorzugt | ausreichend |

**Warum goolm und nicht vodozemac:**
Auf Windows braucht vodozemac CGO (MinGW/GCC Toolchain). Im Dev-Stack (devstack.ps1)
nicht praktikabel. goolm ist funktional identisch fuer Olm+Megolm.
Fuer Production auf Linux: Portierungsplan sieht vodozemac via CGO vor
(siehe `10-portierung.md`). PQXDH-Migration ist Future-Item — siehe `FUTURE_IDEAS.md`.

### Abhaengigkeiten (go.mod)

```go
require (

  modernc.org/sqlite             v1.34.4   // Pure-Go SQLite (kein CGO)
  go.mau.fi/util                 latest
)
```

### Crypto-Store (SQLite)

```go
// internal/crypto/machine.go
import (
  "database/sql"
  _ "modernc.org/sqlite"         // Pure-Go SQLite
  "go.mau.fi/util/dbutil"
  "maunium.net/go/mautrix/crypto"
)

rawDB, _ := sql.Open("sqlite", dbPath+"?_foreign_keys=on&_txlock=immediate")
rawDB.SetMaxOpenConns(1)         // SQLite: single writer
db, _ := dbutil.NewWithDB(rawDB, "sqlite3")

cryptoStore := crypto.NewSQLCryptoStore(
  db,
  dbutil.NoopLogger,
  client.UserID.String(),
  "APPSERVICE",
  pickleKey,
)
```

### OlmMachine Setup

```go
olmMachine := crypto.NewOlmMachine(client, &zlog, cryptoStore, stateStore)
olmMachine.SendKeysMinTrust = id.TrustStateUnset  // sendet Keys an alle Geraete
olmMachine.Load(ctx)
olmMachine.ShareKeys(ctx, -1)
```

**Hinweis zu `SendKeysMinTrust = TrustStateUnset`:**
Standardmaessig wuerde OlmMachine Megolm-Keys nur an verifizierte Geraete senden.
`TrustStateUnset` erlaubt das Senden an alle Geraete — noetig, da der Go Appservice
selbst Cross-Signing-verifiziert ist, aber nicht alle Peers aktiv verifiziert.

## Cross-Signing Bootstrap (`ensureCrossSigning()`)

Implementiert in `internal/crypto/machine.go`. Wird nach `olmMachine.Load()` aufgerufen.

**Erster Start (keine Seeds-Datei vorhanden):**
```go
// 1. Cross-Signing-Keys generieren und hochladen
err := olmMachine.GenerateAndUploadCrossSigningKeys(ctx, nil, "")
// 2. Eigenes Geraet signieren
err = olmMachine.SignOwnDevice(ctx)
// 3. Eigenen Master Key signieren
err = olmMachine.SignOwnMasterKey(ctx)
// 4. Seeds persistieren
seeds := CrossSigningSeeds{
  MasterKey:      olmMachine.GetCrossSigningKeys().MasterKey.Seed,
  SelfSigningKey: olmMachine.GetCrossSigningKeys().SelfSigningKey.Seed,
  UserSigningKey: olmMachine.GetCrossSigningKeys().UserSigningKey.Seed,
}
writeJSON("<dbDir>/cross_signing_seeds.json", seeds, 0o600)
```

**Neustart (Seeds-Datei existiert):**
```go
seeds := readJSON("<dbDir>/cross_signing_seeds.json")
err := olmMachine.ImportCrossSigningKeys(seeds)
// Signaturen erneuern (idempotent)
err = olmMachine.SignOwnDevice(ctx)
err = olmMachine.SignOwnMasterKey(ctx)
```

### Seeds-Datei

- **Pfad:** `<dbDir>/cross_signing_seeds.json`
- **Berechtigungen:** 0o600 (nur Owner lesbar, niemals committen)
- **Inhalt:** Master Key Seed, Self-Signing Key Seed, User-Signing Key Seed (Base64)
- Die Seeds sind dauerhaft — Cross-Signing-Identitaet bleibt ueber Neustarts erhalten
- Verlust = Identitaetsverlust (alle Verifikationen muessen neu durchgefuehrt werden)

## E2EE Rollout Strategie

### Phase 1 — Dev/Testing (abgeschlossen)
- Go Appservice mit `MATRIX_E2EE_ENABLED=false`
- Kein E2EE-Setup noetig
- Fokus auf Funktionalitaet

### Phase 2 — Interne Tests (abgeschlossen)
- `MATRIX_E2EE_ENABLED=true` im Devstack
- Go Appservice: goolm + SQLite Crypto Store
- Python: Klartext via NATS (keine Aenderung)
- Browser: Sendet noch unverschluesselt (vor `initRustCrypto`)

### Phase 3 — Cross-Signing aktiv (aktueller Stand)
- Browser: `initRustCrypto()` aktiv (WASM vodozemac) ✅
- Go Appservice: `ensureCrossSigning()` Bootstrap ✅
- Browser: Cross-Signing via `CrossSigningSetup`-Komponente (QR + SAS) ✅
- MSC4153: Go Appservice ist cross-signing-verifiziert ✅
- MSC4381 Privacy: `sender_key`/`device_id` werden NICHT mitgesendet ✅

### Phase 4 — Production Hardening (siehe `FUTURE_IDEAS.md`)
- `globalBlacklistUnverifiedDevices = true` im Browser
- Megolm Key Backup (C-8) — neue Geraete koennen alte Nachrichten lesen
- PQXDH via vodozemac im Go Appservice (nach Portierung auf Production Linux)
- Authenticated Media (MSC3916) — `allow_legacy_media` Workaround entfernen

## MSC4153 — Implementiert

Ab April 2026 senden major Clients (Element Web, Element X) E2EE To-Device Nachrichten
nur noch an **kryptografisch verifizierte Geraete** (Cross-Signing).

**Status:** Implementiert (C-3)
- Go Appservice: `ensureCrossSigning()` sorgt fuer automatischen Bootstrap bei erstem Start
- Seeds werden persistiert → Cross-Signing-Identitaet ueberlebt Neustarts
- `olmMachine.SendKeysMinTrust = id.TrustStateUnset` → Keys werden an alle Geraete gesendet

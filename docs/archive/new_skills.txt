Wie muss man einen Skill erstellen?

Die kürzeste korrekte Antwort ist:

> Ein Skill ist ein Ordner mit einer SKILL.md-Datei, die YAML-Frontmatter plus Markdown-Instruktionen enthält; optional kommen scripts/, references/ und assets/ dazu. 



Minimalstruktur

my-skill/
├── SKILL.md
├── scripts/      # optional
├── references/   # optional
└── assets/       # optional

Das wird sowohl in den offiziellen Claude/Codex-Dokus als auch im skill-creator-Beispiel so beschrieben. 

Minimaler SKILL.md

---
name: my-skill-name
description: Was der Skill macht und wann er verwendet werden soll
---

# My Skill Name

## Instructions
Klare Schritt-für-Schritt-Anweisungen

## Examples
Ein paar konkrete Beispiele

## Guidelines
Wichtige Regeln und Grenzen

Die offiziellen Docs sagen explizit, dass name und description Pflichtfelder sind. Die description soll sowohl beschreiben, was der Skill tut, als auch wann er triggern soll. 

Was sagt die Studie bzw. das Skill-Repo konkret zur Erstellung?

Das Spannende ist: Das skill-creator-Beispiel im Repo ist fast schon eine kleine Methodik.

Dort wird der Ablauf grob so beschrieben:

1. Intent erfassen
Was soll der Skill können, wann soll er triggern, welches Output-Format ist erwartet? 


2. Früh Edge Cases klären
Input-/Output-Formate, Abhängigkeiten, Erfolgskriterien, typische Beispiele. 


3. SKILL.md schreiben
Besonders wichtig: die Beschreibung muss trigger-stark sein, weil Skills sonst eher untertriggern. Das Repo empfiehlt ausdrücklich, in die Beschreibung konkrete Nutzungskontexte hineinzuschreiben. 


4. Tests schreiben
Das Beispiel empfiehlt 2–3 realistische Test-Prompts und speichert sie in evals/evals.json. 


5. Mit und ohne Skill vergleichen
Also nicht nur „funktioniert?“, sondern verbessert der Skill wirklich etwas gegenüber Baseline? 


6. Iterativ verfeinern
Genau das passt auch zur Studie: Skills helfen unter realistischen Bedingungen oft erst dann mehr, wenn Retrieval und task-spezifische Verfeinerung sauber gemacht werden. 



Was ist laut Studie/Repo der häufigste Fehler?

Der wichtigste praktische Punkt ist wahrscheinlich dieser:

> Nicht die langen Instruktionen sind der Flaschenhals, sondern oft die Trigger-Beschreibung und die Passung zum Task.



Das sieht man indirekt an zwei Stellen:

Die Studie zeigt, dass Nutzen stark sinkt, wenn Skills aus einem großen Pool erst gefunden und angepasst werden müssen. 

Das skill-creator-Beispiel betont deshalb, dass die description klar sagen muss, wann der Skill benutzt werden soll.

# Research

## Local Z Reference

Derived from `Z_Tool_very interessting Quarkdown.md`.

## Working Judgement

Quarkdown is interesting because it is text-based, declarative, buildable and
agent-friendly. It should still be treated as an evaluated renderer, not as a
hard dependency until local builds, packaging and Matrix publication are proven.

## Source Check 2026-04-29

- Quarkdown presents itself as Markdown-based typesetting for documents,
  reports, books and presentations, with HTML/PDF/plain text-style output lanes.
- Its GitHub project and wiki show a young but active ecosystem. That is a good
  experimental fit, but Matrix should keep a fallback renderer.
- For agents, the key advantage is reproducibility: generated source plus data
  and citations can be reviewed, rebuilt and diffed.
- For Matrix clients, the safe default is link/attachment publication. Inline
  interactive rendering belongs to Feature 030.
- Current Quarkdown docs describe HTML and PDF output, with PDF layered on HTML
  post-processing. Matrix therefore treats it as an experimental renderer and
  keeps a deterministic Markdown fallback for static gates.

## Design Consequence

Start with report artifacts, not chat blobs:

```text
agent -> report source + manifest + data/citations -> renderer -> artifact
  -> Matrix link/attachment -> Meta-Harness grounding score
```

The source can be generated by any LLM provider or deterministic pipeline.

## 2026-04-29 Benchmark Handoff

Feature 022 now includes `report-grounding-manifest-001` as a deterministic
RAG benchmark case. It comes from the Quarkdown/Z report-publishing thread but
stays renderer-agnostic: a report citation must carry `report_manifest_id`,
`report_output_path`, `report_renderer`, source artifact metadata and an exact
reference citation.

This gives Feature 027 a static grounding gate before Quarkdown or any other
renderer is promoted in live Matrix chat.

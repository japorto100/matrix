# Feature Review 001-023

Purpose: classify every feature after reading the feature docs and checking the
current code layout. This is not a closeout certificate. It separates:

- `closed/docs`: no active implementation expected.
- `static-done-live-open`: code/static tests look implemented; needs live stack
  proof.
- `implementation-open`: real code/contracts still missing or stubbed.
- `research-needed`: current web/arXiv/project review needed before deciding.

## Summary

`001` is the only feature that is effectively closed as a baseline/document
feature. `002-014` are not all done: several are static-complete but live-open,
and `007/010/012/013/014` still have real implementation or integration gaps.
`015-023` are intentionally active work, with `016/019/021/022/023` carrying
the current agent/RAG/KG/ingestion optimization frontier.

  for `agent`, `control`, `files`, `matrix`, `memory`.

  proxy handlers under `go-appservice/internal/*`.
- Python agent has LangGraph and graphless runners, tool registry, memory,
  control APIs, security, sandbox, scheduler, A2UI, A2A and tracing under
  `python-backend/agent/*`.
- Memory/RAG/KG/ingestion are real Python packages, but still include stubs and
  fallbacks: `memory_fusion`, `memory_engine`, `retrieval`, `ingestion`,
  `kg_pipeline`.
- Control and Memory frontend still have mock fallbacks that must be audited
  before any feature using those tabs is called closed.

## Immediate Review Consequence

Do not treat `implementation_done` in early feature frontmatter as final. For
this repo it often means "static implementation exists" and not "live verified".
The working definition should be:

- close only after gates in `live-verify.md` have evidence, or
- explicitly mark the remaining gate as deferred/out-of-scope with an owner and
  reason.
